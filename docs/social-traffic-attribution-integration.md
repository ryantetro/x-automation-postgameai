# Social Traffic Attribution Integration

This document is the copyable implementation spec for destination sites that receive traffic from the tracked social posts generated by this repo.

## Goal

Track which social posts generate:

- redirect clicks
- landing visits
- sessions
- signups
- demos
- purchases or subscriptions

This repo is the redirect and attribution source of truth. Destination repos are responsible for recording actual on-site visits and conversion events in PostHog.

## Incoming URL Contract

Tracked visitors will arrive with these query params:

- `tracking_id`
- `run_id`
- `platform`
- `campaign_slug`
- `post_id`
- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_content`
- `utm_term`
- `post_sport`
- `post_source`

## Required Destination Behavior

On first page load:

1. Detect the attribution params.
2. Persist them in first-party storage for 30 days.
3. Initialize PostHog.
4. Emit `post_landing` exactly once per landing.
5. Include the same attribution properties on downstream tracked events.
6. Preserve attribution when the user signs up or converts.

## Attribution Object

```ts
export type SocialAttribution = {
  trackingId: string;
  runId?: string;
  platform?: "x" | "threads";
  campaignSlug?: string;
  postId?: string;
  utmSource?: string;
  utmMedium?: string;
  utmCampaign?: string;
  utmContent?: string;
  utmTerm?: string;
  postSport?: string;
  postSource?: string;
  landedAt: string;
  landingPath: string;
};
```

## Required PostHog Events

- `post_landing`
- `page_view_attributed`
- `signup_started`
- `signup_completed`
- `demo_booked`
- `trial_started`
- `checkout_started`
- `purchase_completed`

## Required Event Properties

Every event above must include:

```ts
{
  tracking_id,
  run_id,
  platform,
  campaign_slug,
  post_id,
  utm_source,
  utm_medium,
  utm_campaign,
  utm_content,
  utm_term,
  post_sport,
  post_source,
  landing_path,
  referrer,
  anonymous_distinct_id
}
```

If the user is authenticated, also include your app-level user ID.

## Next.js Reference Implementation

### 1. Create an attribution helper

Add a small client-side module under your analytics layer that:

- reads URL params from `window.location.search`
- builds a `SocialAttribution`
- persists it in both a cookie and `localStorage`
- returns the latest stored attribution

Recommended public helpers:

```ts
captureSocialLanding(): void
getStoredAttribution(): SocialAttribution | null
trackAttributedEvent(eventName: string, props?: Record<string, unknown>): void
```

### 2. Call it from app boot

Call `captureSocialLanding()` once in your root layout/provider on the client.

### 3. Wrap conversion tracking

Replace direct PostHog conversion calls with `trackAttributedEvent(...)` so signups, demos, trials, and purchases always carry the same attribution payload.

## Example Flow

1. User clicks a tracked post link.
2. This repo records the click and redirects to your site with `tracking_id` and UTM params.
3. Your site stores that attribution and emits `post_landing`.
4. User browses and later signs up.
5. Your `signup_completed` event includes the same `tracking_id`.
6. This repo queries PostHog by `tracking_id` and attributes that signup back to the correct post and platform.

## Storage Defaults

- First-touch storage window: 30 days
- Storage method: cookie plus `localStorage` fallback
- Attribution override rule: replace stored attribution only when a new tracked social landing arrives
- Default primary conversion: `signup_completed`

## Success Criteria

After implementation:

- tracked redirects create `post_landing` events on the destination site
- signups and demos can be grouped by `tracking_id`
- this repo can compare X vs Threads traffic and conversions
- dual-published posts stay split by platform instead of collapsing into one traffic bucket
